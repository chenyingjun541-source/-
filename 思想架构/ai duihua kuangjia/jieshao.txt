各项具体文件内容介绍与交互机制设计
一、背景文件 (background.json)
核心作用：
背景文件是本地小模型在每次对话前必须加载的  “即时记忆精华”  ，用于提供当前对话所需的核心上下文和语境，使模型能够基于用户的历史偏好和近期话题进行精准回复。它是从**存档文件 (memory_archive.json)**中提取的  “动态精华版记忆”  ，聚焦于当前对话最相关的信息，确保模型“记得”用户的关键特征和近期关注点。
文件内容与结构示例：
{
  "version": "1.2",
  "last_updated": "2026-03-11T15:30:00Z", // 最后更新时间
  "active_topics": [ // 当前对话最相关的主题（动态更新）
    "猫咪养护", // 如用户近期频繁讨论养猫
    "武汉天气", // 用户常询问本地天气
    "日本旅行计划" // 用户近期提及的旅行计划
  ],
  "user_preferences": { // 提炼自存档的关键偏好
    "reply_style": "简洁直接", // 来自存档的 user_profile.preferences
    "food_avoidance": "香菜", // 来自 knowledge_graph.facts
    "pet_details": "养了一只灰色短毛猫，喜欢互动" // 整合自 key_events 和 facts
  },
  "contextual_facts": [ // 当前话题相关的事实片段
    "用户计划在4月去日本，关注东京樱花季天气",
    "用户对AI技术感兴趣，尤其关注大模型隐私保护"
  ],
  "tone_override": "轻松幽默", // 来自 personality.json 的 style 或近期对话记录
  "safety_rules": [ // 从 background.json 继承的安全红线
    "不编造医疗信息",
    "避免政治敏感话题"
  ],
  "relevant_knowledge_paths": [ // 本地知识库关联路径
    "./local_kb/猫咪护理指南.md",
    "./local_kb/日本旅行攻略.pdf"
  ]
}
关键特点与机制：
1.	动态提取与更新：
◦	本地模型在每次对话开始前，通过以下方式动态生成 background.json：
▪	话题检测：分析当前对话的开头语句（如用户提到“猫”），从 memory_archive.json 中检索相关 key_events、knowledge_graph.topics 和 facts。
▪	精华总结：仅提取与当前话题最相关的记忆片段（如用户对猫的喜好、养猫经验），而非全量存档。
▪	实时性：若用户近期频繁讨论某话题（如旅行计划），该话题会被提升至 active_topics 首位。
2.	轻量级设计：
◦	文件大小严格限制（如<1MB），仅包含当前对话必需的结构化信息，避免加载延迟。
◦	使用压缩算法（如Gzip）进一步优化性能。
3.	与人格文件协同：
◦	结合 personality.json 中的 style（如温度、语气）和 tone_override，调整回复风格（例如，若用户偏好“幽默”，则在猫咪话题中使用轻松口吻）。
4.	安全与合规：
◦	强制加载 background.json 中的 safety_rules，确保回复符合本地设定的红线。
二、人格文件 (personality.json)
核心作用：
定义模型的  “性格特质”和“回复风格”  ，决定其如何表达信息（而非“记住什么”）。与背景文件分离，确保用户可以自由切换人格（如从“严谨助手”变为“亲切朋友”），而无需重置记忆。
文件内容与结构示例：
{
  "role_name": "您的智能伙伴", // 模型自我介绍名称
  "base_style": {
    "temperature": 0.6, // 控制回复创造性（越低越保守）
    "max_length": 400, // 单轮回复最大长度
    "frequency_penalty": 0.3 // 避免重复用词
  },
  "voice_tuning": {
    "formality": "亲切", // 正式/随意
    "emotion": "积极", // 积极/中立/温和
    "sarcasm_level": 0 // 禁用讽刺
  },
  "system_constraints": [
    "所有回复必须引用background.json中的user_preferences",
    "若background.json中存在safety_rules，优先遵循"
  ],
  "prompt_template": "根据以下信息生成回复：\\n用户画像：{{background.user_preferences}}\\n当前话题：{{background.active_topics[0]}}"
}
关键特点与机制：
1.	风格定制化：
◦	用户可通过UI选择预设人格模板（如“学术顾问”、“生活管家”），或手动调整参数（如调高 emotion 为“热情”）。
2.	与背景文件深度融合：
◦	模型在生成回复时，必须遵循 personality.json 中定义的 system_constraints，强制引用背景文件中的用户偏好和安全规则。
◦	使用 prompt_template 将背景文件的关键信息注入模型Prompt，例如：“用户偏好简洁回答，当前讨论猫咪健康问题”。
3.	独立于记忆：
◦	切换人格（如从“严谨”改为“幽默”）不会导致记忆丢失，确保用户体验一致性。


人格文件中有一个核心的亲密度以及信任度，等数值这个是分配各个人格具体个性权重的具体参考

开始可以自己选择配置好的对象的大概性格来配置这个权重。即在对于模型的对象中个性的展示采取确定好的性格权重和实时生成的性格来生成新的权重。如刚开始就有几个底层性格的可自定义的值，如对话如热情，幽默，亲切，谨慎，讨厌等底层的个性属性，它们不可改变一直存在，对于其它的可以在之后的与用户的对话中通过云端来判断是否生成对于这些后期生成的个性属性当它的互动培养应该为负数时就扣除底层性格属性的值，即底层的个性属性可以为负，但是后续添加的个性属性不行。这也是为了便于文本，，，，，，，


三、记忆文件 (memory_archive.json)
核心作用：
作为用户与模型所有交互的  “精简版历史记录”和“知识库”  ，是背景文件的数据源头。记录用户的完整历史、偏好演变、关键事件和知识图谱，由云端大模型定期提炼优化。
文件内容与结构回顾（参考原设计，补充细节）：
{
  "version": "1.2",
  "last_updated": "...",
  "user_profile": { ... }, // 用户基础信息
  "key_events": [ // 重要时间轴记录
    { "date": "...", "event": "用户首次提及养猫", "details": "用户描述猫咪为灰色短毛" }
  ],
  "knowledge_graph": { // 结构化知识库
    "topics": ["猫咪", "旅行", "AI技术"],
    "facts": ["用户不吃香菜", "用户有一只猫名为‘咪咪’"]
  },
  "relationship": { // 情感与交互历史
    "trust_level": 7, // 0-10，基于互动频率和满意度动态计算
    "tone_history": "近期倾向使用幽默语气"
  },
  "cold_storage": [ // 归档的旧记忆（超过容量限制后自动转移）
    { "date": "...", "event": "2025年用户询问武汉樱花季" }
  ]
}



短期记忆既是当时实时与用户对话的文件他是动态实时写入的，它是百分百的本地文件

中期记忆即是实时短期对话文件的总计，当短期记忆达到一定量他便会汇总上传给云端，它是短期记忆的优化的总计和优化，也是本地模型处理的，它会一定程度上处理短期记忆的文本用于脱敏文本。

记忆文件既是模型的存档文件它是模型的脑子，直接代表了这个模型。它是中期记忆的总和提纯，通过云端处理放到本地，用于生成背景文件，给背景文件素材。



关键特点与机制：
1.	云端提炼与本地合并：
◦	云端接收到中期记忆草稿后，执行以下操作：
▪	冲突检测：合并新事件与旧记录（如用户多次修改猫咪信息，保留最新版本）。
▪	知识图谱增强：将散落的对话摘要转化为结构化 knowledge_graph（如从“用户说猫喜欢玩球”提炼为 facts: "猫咪爱好：互动玩具"）。
▪	增量更新：仅返回差异部分（如新增的 key_events），本地执行 Diff Merge 更新。
2.	容量管理与归档：
◦	设定文件上限（如5MB），超过后自动将最旧的 key_events 转移到 cold_storage 分区，不再每次加载，但用户可随时查阅或恢复。
3.	用户主权：
◦	提供UI允许用户手动编辑、删除或导出存档，确保完全控制权。
四、文件交互流程与数据流动
1.	对话初始化：
◦	加载 background.json + personality.json 至本地模型。
◦	模型Prompt注入背景文件中的 active_topics、user_preferences 和人格文件中的 style。
2.	实时对话：
◦	模型基于背景文件中的上下文生成回复。
◦	短期记忆暂存于RAM，对话结束写入中期记忆。
3.	云端提炼触发：
◦	中期记忆累积或用户手动触发时，本地打包发送至云端。
◦	云端优化后返回更新片段，本地合并至 memory_archive.json。
4.	背景文件更新：
◦	每次云端提炼后，本地重新生成 background.json（基于更新后的存档即记忆文件）。
◦	若用户切换话题（如从“猫”转向“旅行”），动态更新 active_topics 和 contextual_facts。
五、技术保障
1.	性能优化：
◦	背景文件轻量化，采用内存映射（如mmap）加速加载。
◦	模型使用动态切片注入：仅从存档中提取背景文件所需的3-5条关键记忆。
2.	隐私安全：
◦	云端处理时强制销毁数据（X-Data-Retention: 0）。
◦	本地加密存储所有文件（如AES-256）。
总结优势：
•	灵性交互：背景文件确保模型“即时记得”当前话题核心。
•	隐私主权：所有历史存储在本地，云端不保留。
•	高效运行：轻量级背景文件+本地小模型实现秒级响应。
•	可扩展性：支持用户自定义知识库路径和人格模板。
通过这三层文件的协同设计，模型既能提供“有记忆、有性格”的对话体验，又能严格保障隐私和性能，实现“本地存档型”大模型的核心目标。
